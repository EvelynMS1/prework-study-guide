Git

  • git is a version control system, it facilitates the tracking of changes made to a web app
  • the manner in which the communication between the doc and git is by using git commands which differ from bash commands.
  • git and git hub have a system in which changes are made to the document and how it is tracked
  • branch is a term that is used and has the function of seperating from the main branch of the document in order to isolate the changes that are currently being made, this ensures that there isnt hiccups that can harm the main branch
  • merge is completed after a pull request, the commit the is pulled in(merged) from the branch that was created for the change and then deployed into the main branch with the end result being displayed on site.
  • Github issues are a sub section of github that helps keep the git hub system organized and on track with changes as well as for monotoring progress more like a built in to do. these github issues are written with markdown language.
  • README's are included via github and are also written in markup language it usually inclueds the user story usually a description for the format of first person perspective of the user who will utilizing the web app including there expectations of the web app followed by the acceptance criteria which is a list of requirements that are needed in order to provide the final product written is as action taken "when" followed by "action observed" then
  • Github is capable of expanding the capabilities of the integrated development environment(IDE- Gives us access to use code on our local machine and by syncing from our cloud based Github) with the usage of extensions
  • Github inclues -source code editor, local build automation, debugger
  • Github Workflow is the process of creating changes and embedding them into default working branch
  • merging is the process of adding our updated edits from our branch to the default branch
  • commiting and pushing changes and edits are ideal and highly recommmended for each and every change mainly bc it allows for the opportubity to go back to that specific state in the change process and make edits , while in the case of saving after multiple changes will have to be modified as a whole if changes need to be made. The other benefit of commiting and and pushing changes is that it allows for back up of work to remote storage meaning that the data is accessible from any device. Also allows for others to make changes and answer questions as well as suggestions
  • pull requests - asks for the input and feedback from peers which is very crucial to the approval of merging the pull requests, once a pull request has been approved there is a follow up description of the change that was added to the default branch can include images links and tables to convey the explanation. finally the branch with the recently approved change must be deleted once in default branch for the prevention of confusion that may arise, take note that the one is still able to revert bak to old branch history if needed.
  • pushing is only pushing to our feature branch aka not default branch, therfore to the github repo
